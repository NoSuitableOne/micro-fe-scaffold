# 微前端框架必备特性梳理

> 本文档用于微前端框架探索项目的基线定义，梳理一个「可用」的微前端框架应具备的核心能力，并区分必备项与增强项，便于后续选型对比与自研方案设计。

## 1. 什么是微前端

微前端（Micro Frontends）是将后端「微服务」思想引入前端：由**主应用（基座 / Shell）**与多个**子应用（微应用）**组成，各子应用可独立开发、独立构建、独立部署，在运行时由主应用统一编排与集成。

一个微前端框架的价值，不在于「能加载多个页面」，而在于能否在**多团队、多技术栈、多发布节奏**下，仍保证整体系统的**一致性、隔离性、可维护性**。

---

## 2. 必备特性总览

| 类别 | 必备特性 | 一句话说明 |
|------|----------|------------|
| 应用编排 | 子应用注册与生命周期 | 主应用能注册子应用，并在适当时机挂载/卸载 |
| 应用编排 | 动态加载 | 按路由或条件按需加载子应用资源 |
| 路由 | 路由分发与激活 | URL 变化时正确激活对应子应用 |
| 隔离 | JS 运行时隔离 | 避免全局变量、副作用污染 |
| 隔离 | 样式隔离 | 避免 CSS 选择器冲突与样式泄漏 |
| 通信 | 跨应用通信机制 | 主应用与子应用、子应用之间可传递信息 |
| 工程 | 独立构建与部署 | 子应用不依赖主应用构建即可发布 |
| 工程 | 资源路径与公共路径 | 子应用静态资源在任意部署路径下可正确加载 |
| 稳定性 | 错误隔离与降级 | 单个子应用异常不应拖垮整个系统 |
| 集成 | 统一接入规范 | 子应用遵循统一 bootstrap/mount/unmount 契约 |

以下逐项展开。

## 3. 必备特性详解

### 3.1 子应用注册与生命周期管理

**为什么必须：** 主应用需要知道「有哪些子应用、何时启用、何时销毁」，否则无法做运行时编排。

**核心能力：**

- **注册（register）**：声明子应用名称、入口地址、激活规则、容器等元信息
- **bootstrap**：子应用首次加载时的初始化（仅执行一次）
- **mount**：子应用进入可见/可用状态，渲染到指定 DOM 容器
- **unmount**：子应用离开视图，释放 DOM、事件、定时器等资源
- **update（可选但常见）**：主应用向子应用传递新 props 时的更新钩子

**典型实现参考：**

- single-spa：`registerApplication` + 标准 lifecycle exports
- qiankun：基于 single-spa，封装 `registerMicroApps` + `start`

**验收标准：**

- [ ] 可在不刷新页面的情况下切换子应用
- [ ] 重复进入同一路由不会重复 bootstrap（或能可控地重新 bootstrap）
- [ ] unmount 后无内存泄漏、无残留事件监听

---

### 3.2 子应用动态加载

**为什么必须：** 微前端的核心收益之一是按需加载，避免首屏加载全部子应用代码。

**核心能力：**

- 根据入口配置（URL / manifest / Module Federation remote）拉取 JS、CSS
- 支持 **HTML Entry**（从 `index.html` 解析脚本）或 **JS Entry**（直接加载入口模块）
- 加载过程可展示 loading 状态
- 支持加载失败重试或降级

**关键设计点：**

| 方案 | 说明 | 代表 |
|------|------|------|
| HTML Entry | 解析子应用 HTML，提取并执行 script/link | qiankun |
| JS Entry | 直接 import 子应用导出的 lifecycle | single-spa + 自定义 loader |
| Module Federation | 运行时共享模块，Webpack/Rspack 原生支持 | Webpack MF、Module Federation 2.0 |
| ESM Import Maps | 浏览器原生 ESM + import map | 新兴方案 |

**验收标准：**

- [ ] 首次访问某子应用路由时才加载其资源
- [ ] 子应用更新部署后，主应用无需重新发布即可加载新版本（配合缓存策略）

---

### 3.3 路由分发与激活规则

**为什么必须：** 用户通过 URL 访问系统，主应用必须将路径映射到正确的子应用。

**核心能力：**

- **激活规则（activeRule）**：基于 path prefix、正则、自定义函数判断子应用是否激活
- **主应用与子应用路由协同**：
  - 主应用路由负责「选哪个子应用」
  - 子应用路由负责「子应用内部页面」
- **浏览器历史模式**：支持 `history` / `hash` 模式
- **404 / 兜底路由**：未匹配时展示默认页或错误页

**常见模式：**

```
https://example.com/           → 主应用首页
https://example.com/app-a/list → 激活子应用 A，A 内部路由 /list
https://example.com/app-b/     → 激活子应用 B
```

**验收标准：**

- [ ] 前进/后退浏览器按钮行为正确
- [ ] 刷新页面后仍处于正确子应用与页面
- [ ] 多个子应用不会同时挂载到同一容器（除非刻意设计）

---

### 3.4 JS 运行时隔离（沙箱）

**为什么必须：** 多个子应用可能来自不同团队、不同技术栈、不同版本，若共享同一 window 全局作用域，极易互相污染。

**核心能力：**

- 隔离或代理 `window` 上的可变全局属性
- 隔离 `document`、`localStorage`（按需）等共享对象的部分操作
- 子应用卸载后恢复被修改的全局状态

**常见方案：**

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| Snapshot Sandbox | 保存/恢复 window 快照 | 实现简单，不支持多实例并行 |
| Proxy Sandbox | Proxy 代理 fakeWindow | 支持多应用，兼容性需考虑 |
| iframe | 天然隔离 | 隔离最强，但通信、路由、体验成本高 |
| 无沙箱（依赖规范） | 约定子应用不使用 window 污染 | 依赖团队纪律，风险高 |

**验收标准：**

- [ ] 子应用 A 修改 `window.xxx` 不影响子应用 B
- [ ] 卸载子应用后全局状态可恢复
- [ ] 第三方 SDK（如埋点、地图）在沙箱内可正常工作或文档说明限制

---

### 3.5 样式隔离

**为什么必须：** 全局 CSS、CSS Modules 失效场景、UI 库样式重置等都会导致样式串扰。

**核心能力：**

- 子应用样式仅作用于其容器范围
- 主应用样式不意外覆盖子应用（或可控共享主题）
- 卸载子应用时移除其注入的 style/link 标签

**常见方案：**

| 方案 | 说明 |
|------|------|
| CSS Scoping（属性选择器） | 为子应用根节点添加唯一属性，改写 CSS 选择器 |
| Shadow DOM | 强隔离，但弹层、Portal 处理复杂 |
| CSS Modules / CSS-in-JS | 依赖子应用自身工程化，非框架层能力 |
| 命名约定（BEM/前缀） | 软隔离，靠规范 |

**验收标准：**

- [ ] 两个子应用使用相同 class 名时不互相影响
- [ ] Modal/Drawer 等 Portal 组件样式不错乱
- [ ] 切换子应用后无残留 style 标签

---

### 3.6 跨应用通信

**为什么必须：** 子应用之间 rarely 完全独立，通常需要共享用户信息、权限、主题、全局通知等。

**核心能力：**

- **主 → 子**：通过 mount props 传递数据、回调
- **子 → 主**：通过回调或事件通知主应用
- **子 ↔ 子**：通过全局事件总线、共享状态容器实现

**常见模式：**

| 模式 | 适用场景 |
|------|----------|
| Props 下发 | 初始配置、用户信息、权限 |
| CustomEvent / EventBus | 低频、解耦的事件通知 |
| 全局 Store（Pinia/Redux/自建） | 需要响应式共享状态 |
| URL / Query | 简单、可分享的状态 |
| Backend API | 权威数据源 |

**设计原则：**

- 优先**单向数据流**，避免多向同步导致状态混乱
- 明确**状态归属**（哪个应用是 Source of Truth）
- 避免子应用直接 import 另一个子应用模块（破坏独立部署）

**验收标准：**

- [ ] 登录态变更后所有子应用同步
- [ ] 通信 API 有 TypeScript 类型或清晰文档
- [ ] 子应用单独运行时（脱离主应用）仍可开发调试

---

### 3.7 独立构建与独立部署

**为什么必须：** 微前端的首要组织价值是「团队自治」，若子应用发布仍依赖主应用发版，则收益大打折扣。

**核心能力：**

- 每个子应用拥有独立仓库或独立 package（monorepo 内独立 app 也可）
- 独立 CI/CD 流水线
- 独立版本号与发布节奏
- 子应用产物可部署到 CDN / 静态服务器，主应用通过配置引用

**与 monorepo 的关系：**

- Monorepo 是**代码组织**方式，不排斥独立部署
- 关键判断标准：子应用构建产物是否可单独上线

**验收标准：**

- [ ] 修改子应用 A 并发布，不影响子应用 B 的构建
- [ ] 主应用仅通过配置（URL/manifest）关联子应用版本
- [ ] 支持灰度：同一子应用不同用户使用不同版本（进阶）

---

### 3.8 公共路径（publicPath）与资源加载

**为什么必须：** 子应用部署路径不固定（CDN 子目录、多环境域名），资源 404 会导致白屏。

**核心能力：**

- 构建时注入 `publicPath` / `base`
- 运行时动态设置 webpack `__webpack_public_path__` 或等价机制
- 正确处理异步 chunk、字体、图片、worker 等资源
- 主应用加载子应用 HTML Entry 时，能修正其中相对路径

**验收标准：**

- [ ] 子应用部署在 `https://cdn.example.com/app-a/v1.2.3/` 下资源全部可访问
- [ ] 懒加载路由 chunk 路径正确
- [ ] 开发环境与生产环境路径策略一致或有明确切换方式

---

### 3.9 错误隔离与降级

**为什么必须：** 微前端将多个「可能失败的应用」组合在一起，必须防止级联故障。

**核心能力：**

- 子应用加载失败：展示错误占位、重试按钮，不影响其他子应用
- 子应用运行时异常：Error Boundary 捕获，可上报监控
- 超时控制：加载超时自动失败
- 可选：部分子应用不可用时，主应用其余功能仍可用

**验收标准：**

- [ ] 子应用 JS 报错时主应用导航仍可用
- [ ] 错误信息可接入监控（Sentry 等）
- [ ] 用户可从不健康子应用切回其他模块

---

### 3.10 统一接入规范（子应用契约）

**为什么必须：** 没有统一契约，主应用无法以相同方式集成不同团队交付的子应用。

**最小契约示例：**

```javascript
// 子应用需导出（或挂载到 window 上）
export async function bootstrap(props) {}
export async function mount(props) {}
export async function unmount(props) {}
```

**props 常见字段：**

| 字段 | 说明 |
|------|------|
| `container` | 挂载 DOM 节点 |
| `basename` | 子应用路由 base |
| `globalState` | 共享状态读写 |
| `onGlobalStateChange` | 订阅全局状态 |
| `setGlobalState` | 更新全局状态 |
| 自定义业务 props | 租户 ID、主题等 |

**验收标准：**

- [ ] 提供脚手架或模板，新子应用 5 分钟内接入主应用
- [ ] 契约版本化，向后兼容策略明确
- [ ] 文档包含本地联调步骤

## 4. 强烈建议但非「第一天必备」的特性

以下特性在真实生产环境中几乎都会遇到，可在框架 MVP 之后迭代：

| 特性 | 说明 |
|------|------|
| **共享依赖** | 避免 React/Vue 等多份实例；Module Federation shared / externals |
| **预加载** | 空闲时预加载高频子应用，优化切换体验 |
| **Keep-Alive / 缓存** | 切换 tab 后保留子应用状态，避免重复请求 |
| **权限与菜单集成** | 主应用统一菜单，按权限动态注册子应用 |
| **开发联调模式** | 主应用 dev + 子应用 dev 热更新联调 |
| **SSR / SSG** | 首屏 SEO 与性能（实现复杂，多数场景后期才需要） |
| **样式/主题共享** | Design Token、CSS Variables 跨应用一致 |
| **TypeScript 类型共享** | 契约、全局状态的类型定义包 |
| **版本兼容策略** | 主应用与子应用的最低兼容版本矩阵 |
| **可观测性** | 性能指标、加载耗时、子应用健康度 |
| **国际化协调** | 语言切换跨应用同步 |
| **弹层/遮罩协调** | 全局 z-index、Modal 挂载点 |

## 5. 框架选型对比维度

后续对比 qiankun、micro-app、wujie、Module Federation、single-spa 等方案时，建议用以下维度打分：

| 维度 | 考察问题 |
|------|----------|
| 隔离能力 | JS/CSS 隔离方案与边界 case |
| 技术栈无关性 | 是否支持 React/Vue/Angular/原生混用 |
| 接入成本 | 改造现有应用的工作量 |
| 独立部署 | 是否真正解耦发布 |
| 性能 | 首屏、切换耗时、内存占用 |
| 生态与维护 | 社区活跃度、文档、大案例 |
| 调试体验 | 独立运行、联调、错误定位 |
| 浏览器兼容 | 是否需要 Proxy、是否支持 IE |
| 扩展性 | 插件机制、自定义 loader/sandbox |

## 6. 自研框架 MVP 建议范围

若目标是自研或深度封装，建议 MVP 只覆盖**第 3 节全部必备特性**：

```
Phase 1（MVP）
├── 子应用注册 + lifecycle
├── 路由激活 + 动态加载
├── 基础 JS 沙箱 + 样式隔离
├── props 通信 + 简单全局状态
├── publicPath 处理
└── 错误边界 + 加载失败 UI

Phase 2
├── 共享依赖 / Module Federation
├── 预加载 + Keep-Alive
├── 开发联调工具链
└── 监控与性能指标

Phase 3
├── 权限菜单一体化
├── SSR
└── 灰度发布 / 版本管理
```

## 7. 常见反模式（框架设计时需规避）

1. **子应用互相 import**：破坏独立部署，形成隐式耦合
2. **过度共享状态**：全局 Store 变成「前端单体数据库」
3. **无 unmount 清理**：内存泄漏、重复注册路由
4. **主应用承载过多业务**：基座应薄，业务在子应用
5. **忽略 CSS Portal 问题**：弹层样式在「隔离方案」下频繁踩坑
6. **沙箱与第三方脚本冲突**：地图、编辑器、旧 SDK 需白名单机制
7. **路由 base 配置错误**：刷新 404 的最常见原因之一
